# 《Clinoppe 世界》微信小程序游戏 α 版 后端设计文档

**文档版本**：V1\.0
**对应游戏版本**：α 版（2026\.7\.1–2026\.7\.31）
**运行环境**：微信小程序、兼容海外 Web/App 端
**开发技术栈**：PHP 8\.1 \+ ThinkPHP、MySQL 8、Redis、WebSocket、JWT、腾讯 COS\+CDN
**关联 PRD**：《Clinoppe 世界》V1\.0 需求文档

## 一、文档概述

### 1\.1 编写目的

本文档基于游戏 PRD 中**α 版功能需求**，定义后端整体架构、数据库设计、接口规范、业务逻辑、缓存策略、数据存储、权限及部署规则，指导后端开发、联调、测试及交付，保障 α 版全功能正常落地、多端数据互通。

### 1\.2 版本范围与功能边界

#### 1\.2\.1 包含功能

1. 完整寻物关卡系统（5 套地图、关卡解锁、星级评分、进度留存）

2. 拼图玩法（3 档难度、计时 / 进度 / 结算数据）

3. 叠叠乐正式版数据逻辑

4. 海内外通用社交分享（短码、链接、跨渠道分享）

5. 多端用户体系（剥离微信头像依赖、游戏专属头像皮肤）

6. 云端统一数据存储、成就系统、藏品收集系统

7. 道具系统（提示、放大镜、时长延长）、体力系统

8. 月卡权益体系（每日道具礼包、限时无限体力、限定皮肤）

9. 基础排行榜、微信异步互助寻物

10. JWT 身份鉴权、全端数据互通

#### 1\.2\.2 暂不包含功能

双人你藏我找、多人 PK 对战、实时 WebSocket 对战、内购支付通道、完整运营后台、防作弊体系（β 版迭代）。

### 1\.3 运行环境说明

- 服务端系统：Linux CentOS 7\+/Ubuntu

- Web 服务：Nginx

- 编程语言：PHP 8\.1

- 框架：ThinkPHP 6

- 数据库：MySQL 8\.0（主业务数据）

- 缓存：Redis 6\.0\+（会话、排行榜、道具、体力、热点数据）

- 对象存储：腾讯 COS（图片、音效、美术资源）\+ CDN 加速

- 鉴权：JWT

- 通信：HTTP/HTTPS（α 版无实时对战，暂不启用 WebSocket）

## 二、整体架构设计

### 2\.1 架构分层

采用**经典前后端分离架构**，分层清晰、解耦易维护，支持微信小程序、海外 Web/App 多端统一接入：

1. **接入层**：Nginx 反向代理、请求转发、跨域处理、HTTPS 证书、CDN 资源分发

2. **鉴权层**：JWT 令牌校验、身份识别、多端账号统一登录、权限拦截

3. **接口层**：ThinkPHP 控制器，接收前端请求、参数校验、路由分发

4. **业务逻辑层**：服务层（Service），封装游戏核心业务、数据计算、规则判定

5. **数据访问层**：模型层（Model），操作 MySQL、Redis，统一数据读写

6. **存储层**：MySQL（持久化数据）、Redis（缓存数据）、腾讯 COS（静态素材）

### 2\.2 架构特点

1. **多端兼容**：一套后端服务同时支撑微信小程序、海外 Web、App，数据完全互通

2. **数据云端化**：替换 Demo 本地存储，全量用户数据云端持久化

3. **读写分离**：热点数据（体力、道具、排行榜、关卡进度）走 Redis 缓存，核心数据落地 MySQL

4. **无平台绑定**：剥离微信原生接口强依赖，头像、分享、账号体系通用化

5. **高可用**：Redis 缓存减轻数据库压力，接口做异常捕获、日志记录、容错处理

### 2\.3 业务模块划分

后端按业务拆分为 8 大核心模块，模块间低耦合：

1. **用户账号模块**：登录、信息、头像皮肤、多端身份统一

2. **关卡寻物模块**：地图关卡、通关进度、星级评分、对局数据

3. **迷你游戏模块**：叠叠乐、拼图（难度、计时、分数、进度）

4. **道具 \& 体力模块**：道具发放、使用、库存、体力恢复、消耗

5. **月卡权益模块**：月卡状态、每日礼包、无限体力、限定皮肤发放

6. **成就 \& 藏品模块**：成就解锁、藏品收集、状态记录

7. **社交分享模块**：短码生成、分享链接、异步互助、跨渠道分享

8. **排行榜模块**：分数排行、榜单数据统计、缓存更新

## 四、Redis 缓存设计

### 4\.1 缓存使用场景与 Key 规则

统一 Key 前缀：`cl:alpha:`，区分不同业务，设置合理过期时间，避免数据堆积。

|缓存 Key|存储内容|过期时间|用途|
|---|---|---|---|
|cl:alpha:token:\{token\}|JWT 令牌、用户 ID、登录态|7 天|身份鉴权、会话保持|
|cl:alpha:user:\{user\_id\}|用户基础信息|2 小时|高频查询用户资料|
|cl:alpha:stamina:\{user\_id\}|用户体力、恢复计时|30 分钟|体力读写|
|cl:alpha:item:\{user\_id\}|用户道具库存|1 小时|道具查询、扣减|
|cl:alpha:ranking:total|总分数排行榜 Top100|10 分钟|榜单展示|
|cl:alpha:level:list|全量关卡配置|永久|关卡基础配置|
|cl:alpha:share:\{share\_code\}|分享短码关联数据|24 小时|分享跳转、互助|
|cl:alpha:vip:daily:\{user\_id\}|当日月卡礼包领取标记|当日 24 点过期|防重复领取|

### 4\.2 缓存策略

1. **读优先缓存**：前端请求先查 Redis，无数据再查 MySQL，查询后回写缓存

2. **写同步更新**：修改数据（道具、体力、分数、月卡）时，**同时更新 MySQL\+Redis**

3. **定时刷新**：排行榜、全局配置定时全量刷新，保证数据一致性

4. **过期淘汰**：热点短期数据设置过期时间，自动清理无效缓存

## 五、接口规范设计

### 5\.1 基础约定

1. **请求方式**：统一使用 `POST`，参数 JSON 格式提交

2. **接口前缀**：`/api/alpha/`

3. **数据格式**：所有接口返回标准 JSON

4. **身份鉴权**：所有业务接口请求头携带 `Authorization: Bearer {JWT令牌}`

5. **跨域**：服务端配置 CORS，允许微信、Web、App 多端域名访问

6. **字符编码**：UTF\-8

### 5\.2 统一返回格式

```json
{
  "code": 0,       // 状态码：0=成功 非0=错误
  "msg": "操作成功",// 提示文案
  "data": {}       // 业务数据主体
}
```

**常用状态码**：

- 0：成功

- 1001：令牌无效 / 过期（重新登录）

- 1002：参数缺失 / 参数错误

- 1003：体力不足

- 1004：道具不足

- 1005：今日礼包已领取

- 1006：关卡未解锁

## 六、核心业务逻辑说明

### 6\.1 多端账号与头像逻辑

1. 以`union_id`作为**全局唯一账号**，微信、Web、App 共用一套数据

2. 微信端：优先展示微信原生头像，支持切换游戏内头像皮肤

3. 海外 Web/App 端：隐藏微信头像入口，仅可使用游戏专属头像皮肤

4. 头像皮肤解锁来源：初始赠送、月卡、成就、运营奖励，全端互通。

### 6\.2 关卡解锁 \& 星级规则

1. 5 套关卡按顺序解锁，必须通关前置关卡才可开启下一关

2. 单局结算根据得分判定 1/2/3 星，星级数据云端永久留存

3. 重复游玩关卡，最高分实时更新，星级以历史最高为准。

### 6\.3 体力机制

1. 基础规则：拥有固定体力上限，游玩寻物 / 迷你游戏扣除体力

2. 体力自动定时恢复，恢复规则配置在服务端

3. 月卡用户：开启**每日限时无限体力**，当日游玩不扣体力。

### 6\.4 道具使用逻辑

1. 道具分为提示、放大镜、时长延长，单局使用次数前端限制，后端做二次校验防作弊

2. 道具来源：初始赠送、月卡每日礼包、成就奖励

3. 使用道具后端扣减对应库存，数据实时入库 \+ 更新缓存。

### 6\.5 月卡权益逻辑

1. 月卡有效期按自然天数计算，过期自动关闭特权

2. 每日 0 点重置「礼包领取状态」，每位用户每日仅可领取 1 次免费道具礼包
